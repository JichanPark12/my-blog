## 1. 프로젝트 시작과 기술 스택 결정

블로그를 만들기 위해 처음엔 **Gemini** 를 사용하여 아이디어를 구체화했습니다.

복잡한 데이터베이스가 필요 없는 간단한 블로그를 목표로 했기에, **파일 시스템 기반(File System Based)** 아키텍처를 선택했습니다.

### 기술 스택

- **Framework**: Next.js
- **UI Library**: shadcn/ui
- **Data Source**: Local Markdown Files

### CRUD 전략

블로그의 핵심 기능인 CRUD 중에서, 코드로 구현하는 것은 오직 **Read(조회)** 뿐입니다.
나머지 **Create(작성), Update(수정), Delete(삭제)**는 개발자인 제가 직접 Markdown 파일을 추가하고 Git Push를 하는 방식으로, 가장 심플하고 확실한 방법을 택했습니다.

## 2. 체크리스트와 Task.md

개발의 나침반이 될 체크리스트를 요청했습니다.

해당 체크리스트의 완성도를 높이기 위해, 다른 AI들에게도 내용을 공유하며 피드백을 받았습니다.
"이 계획 어때?"라고 물어보고, 괜찮다는 의견과 함께 저도 내용을 검토하며 충분하다고 판단한 뒤, 본격적인 작업을 시작했습니다.

## 3. 초기 설정: AI의 한계와 인간의 개입

Next.js 설치와 shadcn/ui 초기 설정은 AI에게 맡기기에 조금 애매한 부분이 있었습니다.
Next.js의 설치 옵션이 다양하고, shadcn/ui 설정 과정에서 AI가 종종 헤매는 경향이 있기 때문입니다.

따라서 프로젝트의 기반이 되는 **초기 설치와 설정은 제가 직접 진행** 했습니다. 이를 통해 확실한 기반 위에서 AI와의 협업을 시작할 수 있었습니다.

## 4. Vibe Coding: 단계별 구현 프로세스

본격적인 구현은 **체험적 코딩(Vibe Coding)** 방식으로 진행되었습니다.
체크리스트를 하나씩 도장 깨기 하듯 진행하는 방식입니다.
이 과정에서 체크리스트가 수정되거나 추가되는 경우가 많았습니다. 가령 대표적으로 O(1) 조회 성능을 위한 posts-map.json 생성 스크립트를 추가하는 것이 있습니다.

1.  **계획 수립 (Planning)**:
    - 단계별로 AI가 계획을 세우고 저에게 보여줍니다.
    - 계획이 완성되면 체크리스트에 추가됩니다.
2.  **검토 및 수정 (User Review)**:
    - 저는 계획을 검토하고, 추가하거나 빼고 싶은 부분, 수정이 필요한 부분을 지시합니다.
    - 체크리스트가 수정되거나 추가되는 경우가 많았습니다.
3.  **구현 (Implementation)**:
    - 확정된 계획에 따라 AI가 코드를 작성하고 구현합니다.
4.  **코드 확인 (Code Review)**:
    - 구현된 코드를 대충 훑어보고(Skim), 핵심 로직을 매우 상세히 보기 보단 간단히 훑어보며 이상한 점은 없는지 찾았습니다.
    - 구현된 결과가 제 기대치와 일치하지 않으면, AI에게 지시를 내리고 다시 구현하도록 했습니다(병렬 요청이 가능함에도 불구하고 직렬 요청을 하는 부분들을 병렬로 변경을 지시).
    - "오케이(OK)" 사인이 떨어지면 다음 단계로 넘어갑니다.

이러한 반복적인 피드백 루프를 통해, 제가 원하는 방향으로 빠르게 블로그를 완성할 수 있었습니다.
